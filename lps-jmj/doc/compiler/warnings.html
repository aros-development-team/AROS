Compilation Warnings

Definition

A compilation warning is a warning about an error in the input to the compiler.  Compilation warnings are distinct from compilation errors in that the former doesn't prevent the compiler from producing an executable file.

Compilation warnings are represented by instances of CompilationError.  They are distinguished from errors by the fact that they are passed as an argument to a method on CompilationErrorHandler, instead of thrown.  A single compilation can generate multiple compilation warnings.

Validation warnings and errors are compilation warnings.

Server Behavior

http://server/path/to/file.lzx retrieves an HTML file that includes an <object> element for embedding the swf (which is then returned by a subsequent browser request).  When there are compilation warnings, the retrieved page looks like this instead:

 <html>...
   <object..../>
   <h1>Compilation Warnings</h1>
   <pre>compilation warnings, maybe several lines of them</pre>
 </html>

Rationale

The intent is that the easiest thing to type in during development, gets you the right response.  For a deployed app, you've presumably verified that it doesn't have warnings in the deployment setting (otherwise you've got worse probems than the warnings showing up anyway), and you're probably setting the application within an HTML page that you've written, and are requesting the swf directly, instead of requesting the html that embeds the swf, anyway.  More clearly, we define the http://server/path/to/files.lzx as the syntax for executing the application during development, and other syntaxes with more query parameters to execute an application during deployment.

A couple of previous behaviors didn't pan out:

Validation errors were compiler errors, not warnings: a request for an invalid file retrieved a page that listed the errors, instead of an executable.  This was too strict: prevented internal developers from implementing ahead of the schema.

Warnings were reported in the debug window of the compiled application.  This was too lax: it was too easy to miss compilation warnings --- when you're using the server for development (the only way outside developers get to), and if you aren't using <debug/>, you don't get any notification.  At one point bug was submitted that the compiler accepted text inside an element --- actually it generated a warning, but this is swallowed if you aren't using <debug/>.
